## Context: Nymeria's Trading Bot

Nymeria wants to develop a trading bot using a simple grammar to generate logical sentences.
These sentences specify how outputs of weighted mean filters combine to produce a **"Buy" signal**.

### Grammar Definition
Buy        →  Comp_expr Logic_op Buy
Buy        →  Comp_expr
Comp_expr  →  Filter Comparator Filter
Filter     →  Fname(N, T)
Logic_op   →  ∧ | ∨
Comparator →  < | = | >
Fname      →  sma | lma | ema
N          →  1, 2, 3, 4, ...
T          →  i | i−1

### Token Example

`sma(40, i)` — the 40-day **simple moving average** of a series at time point `i`

- `∧` = AND, `∨` = OR
- `i` = current time point, `i−1` = previous time point
- Filter types: `sma` (simple MA), `lma` (linear MA), `ema` (exponential MA)

## Question 1 — Hypothesis Space & Language

### (a) Definitions
- **Hypothesis space**: The set of all valid expressions (programs/rules) that can be generated by the grammar — i.e., all syntactically valid "Buy" signals.
- **A model**: A single expression from the hypothesis space; one specific trading rule that maps filter outputs to a Buy signal.

### (b) Examples
- **In the hypothesis space**: `sma(10, i) > sma(50, i)` — buy when 10-day SMA crosses above 50-day SMA.
- **Not in the hypothesis space**: Something requiring 3+ filters in a single comparison, negation (`¬`), arithmetic on filter outputs (e.g., `sma(10,i) - sma(50,i) > 0`), or nested function calls.

### (c) Size of the Hypothesis Space
The space is **infinite** — `N` is unbounded (1, 2, 3, ...), so there are infinitely many possible filter parameters.

### (d) Golden Cross Expression
A Golden Cross = time point where the 10-day EMA has *just* crossed above the 20-day SMA.
ema(10, i) > sma(20, i) ∧ ema(10, i−1) = sma(20, i−1)
or with `<` for the previous point:
ema(10, i) > sma(20, i) ∧ ema(10, i−1) < sma(20, i−1)

**Is it in the hypothesis space?** Yes — both conjuncts are valid `Comp_expr` terms joined by `∧`, which matches the `Buy → Comp_expr Logic_op Buy` rule.

## Question 2 — Search & Optimisation

### (a) Evaluating Model Quality
- **Backtest** the model on historical price data.
- Measure a fitness metric such as: total return, Sharpe ratio, win rate, or profit factor.
- Fitness = how well the Buy signal would have performed over a historical window.

### (b) Well-Behavedness of the Search Space

The space is **not smooth** — it's a mix of discrete structural choices (grammar rules, comparators, filter types) and integer parameters (N values). This makes it **partially discrete and partially numeric**.

| Method | Applicable? | Why |
|---|---|---|
| **Gradient descent** | ❌ Mostly no | Space is largely discrete/symbolic; gradients undefined for structure |
| **Direct methods** (e.g., Nelder-Mead) | ⚠️ Partially | Could work on the numeric parameters (N) if structure is fixed |
| **Stochastic single-state** (e.g., hill climbing, simulated annealing) | ✅ Yes | Handles discrete/combinatorial spaces naturally via Tweak operators |

### (c) Basic Hill Climbing Pseudocode
function HillClimb(evaluate, tweak, initial):
current ← initial
loop:
candidate ← tweak(current)
if evaluate(candidate) ≥ evaluate(current):
current ← candidate
until termination condition met
return current

### (d) Tweak Operations
Suitable tweaks for Nymeria's models:

- **Change a parameter N** — e.g., `sma(10, i)` → `sma(12, i)`
- **Change a filter type** — e.g., `sma` → `ema`
- **Change the time point T** — `i` → `i−1` or vice versa
- **Change the comparator** — `<` → `>` or `=`
- **Add/remove a conjunct/disjunct** — extend or shorten the Buy chain
- **Swap a sub-expression** — replace one `Comp_expr` with another

**Do tweaks stay in the hypothesis space?**
- Parameter/comparator/type changes: ✅ Yes, always valid
- Structural changes (add/remove clauses): ⚠️ Need care to ensure grammar rules are respected — can be guaranteed if the tweak is grammar-aware

### (e) Hill Climbing vs. Steepest Ascent Hill Climbing

| | Hill Climbing | Steepest Ascent HC |
|---|---|---|
| **How it works** | Accepts first improving neighbour | Evaluates *all* neighbours, picks the best |
| **Speed per iteration** | Fast | Slow (evaluates many candidates) |
| **Quality of move** | May take suboptimal steps | Always takes the locally best step |
| **Compute cost** | Low | High (especially if neighbourhood is large) |
| **Risk** | More erratic path | More principled, but same local optima traps |

**Trade-off summary**: Steepest Ascent is more deliberate but far more expensive. In Nymeria's case, the neighbourhood could be very large (many N values to try), making Steepest Ascent costly. Basic Hill Climbing may be preferred for speed, accepting some noise in the search path.

## Question 3 — Search Space Theory

### (a) Finding the Global Optimum

| Search Space | Can we guarantee finding the optimum? | Why |
|---|---|---|
| **Finite, enumerable** | ✅ Yes | Can enumerate all candidates and evaluate each one |
| **Infinite, enumerable** | ❌ No (in general) | Can enumerate in order, but may never reach the optimum in finite time |
| **Infinite, non-enumerable** | ❌ No | Cannot even list candidates systematically; exhaustive search is impossible |

### (b) Why is Randomness Important in Optimisation?

- **Escaping local optima** — deterministic methods get stuck; randomness allows jumps to new regions
- **Exploration vs. exploitation** — randomness drives exploration of unseen parts of the space
- **Avoiding bias** — removes systematic blind spots in search direction
- **Restarts** — random restarts let hill climbing sample multiple basins of attraction
- **Stochastic acceptance** (e.g., simulated annealing) — probabilistically accepting worse solutions prevents premature convergence
